# V1.0 Release Review

This review covers the FastAPI backend, SQLite persistence layer, hardware transports, static UI serving, tests, and local operations endpoints as of June 21, 2026.

## Current Release Posture

The project has the core v1.0 backend flows in place for timing, queueing, persistence, course management, hardware simulation, leaderboards, settings, and operations. Admin-class mutating endpoints are protected with local admin token auth; public kiosk queue join and read-only display/status endpoints remain unauthenticated for local gym workflows.

## Security Findings

- High: the default admin PIN is `1234` in `config/settings.yaml`. A real deployment must override it through `ADMIN_PIN` or a local secret-management process before enabling admin operations.
- High: admin tokens support `ADMIN_TOKEN_SECRET`, but fall back to the PIN when no token secret is configured. Set `ADMIN_TOKEN_SECRET` for deployment.
- High: admin login uses a bounded in-memory throttle. Multi-process deployments would need a shared rate-limit store, but the current local-first deployment runs as one process.
- Medium: no TLS or network binding guidance is enforced. `host: 0.0.0.0` is practical for gym LAN usage, but deployment docs should require trusted LAN or reverse-proxy protection.
- Medium: generated bytecode, local database files, WAL/SHM files, and logs are ignored for future work. Already tracked generated/runtime files should be removed from version control during the v1.0 cleanup commit.

## Reliability And Exception Handling

- Added consistent JSON handlers for HTTP errors, request validation errors, and unexpected server errors in `app/main.py`.
- Added bounded log tailing so large log files are not read fully into memory.
- Added WebSocket disconnect cleanup in a `finally` block to avoid retaining stale connections after non-disconnect errors.
- Updated logging setup to replace prior app handlers instead of stacking duplicate console/file handlers across repeated app creation.
- Remaining risk: countdown completion uses untracked `asyncio.create_task` calls. Rapid repeated starts can create many sleeping tasks. v1.0 should centralize countdown task ownership and cancellation in the timer service.
- Remaining risk: hardware transports swallow connection failures into status fields. That is acceptable for local operation, but v1.0 should emit system events for repeated hardware failures.

## Memory And Resource Review

- No unbounded in-process data structures were found in normal timer state.
- WebSocket connections are stored in a set and are removed during publish failures and now during all route exits.
- Log retrieval is now bounded by requested tail length.
- Login rate-limit state is bounded by client count and time window.
- SQLite sessions are generally scoped with dependencies or context managers and closed correctly.
